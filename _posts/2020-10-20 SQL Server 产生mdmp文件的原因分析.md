背景说明
    收到前端交付说，有一个现场的MSSQL 2008 R2 日志目录下有大量的文件，很快操作系统的C盘就被打满了。查看后发现log目录下有大量的mdmp，log后缀的文件，自己检查数据库的配置也未发现异常，网上查了大量的资料，说是bug导致，
    建议发给微软分析。后来打补丁sp3后观察，数据库未在dump异常文件，过来两天，早上交付打电话说，那台数据库磁盘又满了，折腾了两天又回到了原点，今天想还是先找研发一起看看，从使用的层面看看是什么地方
    
原因分析：
    安装软件的版本存储过程字段发生了变更，导致存储过程调用失败
    
总结：
    折腾了三天，总算是把问题找到了，解决问题的思路在一开始走了些弯路，总算从数据库层找问题，是不是bug，如何修复，而没有站在使用者的角度思考，导致问题发生的原因，可能是使用的时候方式不对，不用软件对
    异常的处理不一样，
```
mdmp的叫mini dmp，也可以叫memory dmp，这是由于SQL Server 在运行过程中，遇到了一些bug或者错误而进行转储以便记录出错信息的文件。一般对这类文件的处理，都是建议打包后提交给微软分析的。
在无法获得微软帮助的情况，就需要自己对此类文件进行分析了，然后找出问题原因，从而进行解决。
```
